# 📌 KAMIS 가격 수집 및 분석 시스템 (Refactoring & Deep Dive)

## 🧭 프로젝트 소개

본 프로젝트는 공공 Open API(KAMIS)를 기반으로  
전국 농산물 가격 데이터를 수집하고 분석하기 위한 시스템입니다.

기존 실무에서 개발했던 프로젝트를 기반으로,  
단순 기능 구현 수준을 넘어 **설계 개선, 성능 최적화, 안정성 확보**를 목표로  
다시 구조를 정리하고 고도화하는 과정에서 진행한 프로젝트입니다.

👉 **실무 프로젝트를 다시 설계하고 개선한 회고형 프로젝트입니다.**

---

## 🎯 프로젝트 목적

- 단순 데이터 수집 → **운영 가능한 배치 시스템으로 개선**
- 실무에서 겪었던 문제를 기반으로  
  → 재현 → 분석 → 구조 개선
- 핵심 백엔드 역량 강화
  - 배치 설계
  - 데이터 정합성
  - 멱등성
  - 성능 최적화

---

## 🏗️ 기술 스택

- Java 17
- Spring Boot 3.5
- Spring Batch
- Spring Data JPA
- OpenFeign
- MySQL
- Gradle

---

## 🧱 시스템 아키텍처

```text
[외부 KAMIS API]
        ↓
   (Feign Client)
        ↓
 [Raw Data 저장]
        ↓
 [데이터 정제 / 변환]
        ↓
 [Price Data 저장]
        ↓
 [조회 API 제공]
```

### ✔ 설계 포인트

- API 호출 / 저장 / 후처리 **완전 분리**
- Batch 중심 데이터 파이프라인
- 데이터 정합성 보장 구조

---

## 📦 주요 기능

### 1️⃣ 가격 데이터 수집 (Batch)

- KAMIS Open API 기반 데이터 수집
- Spring Batch 기반 일일 배치
- 지역 / 품목 단위 데이터 수집

---

### 2️⃣ 데이터 정규화

- 시장명, 품종, 등급 등 비정형 데이터 처리
- 매핑 테이블 기반 표준화
- 비교 가능한 데이터 구조로 변환

---

### 3️⃣ 가격 조회 API

- 품목 / 지역 / 날짜 기준 조회
- 최신 가격 조회
- 검색 기능 제공

---

### 4️⃣ 배치 관리 기능

- 배치 수동 실행 API
- 배치 실행 이력 조회
- 실패 데이터 재처리

---


## 🧑‍💻 회고

실무에서는 “동작하는 것”이 목표였다면,  
이번 프로젝트에서는 “왜 이렇게 설계해야 하는가”에 집중했습니다.

특히 다음을 깊이 이해할 수 있었습니다.

- 배치 시스템에서 멱등성의 중요성
- 데이터 정합성을 유지하는 설계 방식
- 병렬 처리 시 고려해야 할 요소
- 장애 상황에서도 무너지지 않는 구조

→ 단순 구현을 넘어  
운영 가능한 시스템 설계 경험을 목표로 진행한 프로젝트입니다.
